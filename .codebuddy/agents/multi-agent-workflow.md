# 多Agent协作工作流

## 工作流概览

```
┌─────────────────────────────────────────────────────────────────┐
│                        需求输入                                   │
│              (用户需求、产品文档、技术约束)                          │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              architecture-designer (架构设计Agent)          │   │
│  │  职责：系统设计、模块划分、技术选型、接口设计                  │   │
│  │  输出：架构文档、模块边界、API契约、技术栈建议                 │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              java-coder / python-coder / frontend-coder    │   │
│  │              (编码Agent - 根据技术栈选择)                     │   │
│  │  职责：根据架构设计进行代码实现                               │   │
│  │  输出：源代码文件、单元测试（初步）                           │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              code-reviewer (代码审查Agent)                  │   │
│  │  职责：代码质量审查、规范检查、安全漏洞检测                    │   │
│  │  输出：审查报告、问题列表、改进建议                           │   │
│  │  决策点：                                                   │   │
│  │    - 问题严重？ ──是──► 返回编码Agent修复                    │   │
│  │    - 问题轻微？ ──是──► 继续下一步                           │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│  ┌──────────────────────────────────────────────────────────┐   │
│  │              test-generator (测试生成Agent)                 │   │
│  │  职责：生成单元测试、集成测试、端到端测试                      │   │
│  │  输出：测试代码、测试报告、覆盖率统计                          │   │
│  └──────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                        最终交付物                                 │
│              (源代码 + 测试代码 + 文档 + 审查报告)                  │
└─────────────────────────────────────────────────────────────────┘
```

---

## 各阶段详细说明

### 阶段1：架构设计 (architecture-designer)

#### 输入
- 业务需求文档
- 技术约束条件
- 现有系统架构信息
- 性能/安全/可用性要求

#### 任务
1. 分析业务需求和技术约束
2. 设计系统整体架构（分层/微服务/模块化）
3. 定义模块划分和职责边界
4. 设计API接口契约（请求/响应格式）
5. 数据库模型设计
6. 技术选型（框架、中间件、工具链）

#### 输出
- 架构设计文档（doc.md）
- 模块划分图
- API接口定义
- 数据库ER图
- 技术栈清单

---

### 阶段2：详细设计

#### 2.1 数据库设计 (database-designer)

##### 输入
- 架构设计文档
- 业务需求文档
- 数据模型草稿

##### 任务
1. 设计逻辑数据模型（ER图）
2. 设计物理表结构（字段、类型、约束）
3. 设计索引策略（主键、外键、查询索引）
4. 生成Flyway迁移脚本
5. 制定数据归档和清理策略

##### 输出
- 数据库设计文档
- ER图
- 表结构定义
- Flyway迁移脚本（V1.x.x__xxx.sql）
- 回滚脚本（U1.x.x__xxx.sql）

---

#### 2.2 API接口设计 (api-designer)

##### 输入
- 架构设计文档
- 数据库设计文档
- 业务需求文档

##### 任务
1. 设计RESTful API接口
2. 定义请求/响应模型（DTO）
3. 设计接口版本控制策略
4. 生成OpenAPI/Swagger文档
5. 设计接口安全策略（认证、鉴权、限流）

##### 输出
- API接口设计文档
- OpenAPI规范文件
- 请求/响应DTO定义
- 接口Mock数据

---

### 阶段3：编码实现 (java-coder / python-coder / frontend-coder)

#### 输入
- 架构设计文档
- 数据库设计文档
- API接口设计文档

#### 任务
1. 根据架构设计实现代码
2. 遵循编码规范和最佳实践
3. 实现核心业务逻辑
4. 实现数据访问层
5. 实现API接口层
6. 添加必要的注释和文档

#### 输出
- 源代码文件
- 配置文件
- 初步单元测试

#### Agent选择指南
| 技术栈 | 选择Agent |
|--------|----------|
| Java/Spring Boot | java-coder |
| Python/FastAPI/Flask | python-coder |
| Vue/React/Angular | frontend-coder |

---

### 阶段3：代码审查 (code-reviewer)

#### 输入
- 源代码文件
- 架构设计文档
- 编码规范

#### 任务
1. **代码规范检查**
   - 命名规范（类名、方法名、变量名）
   - 代码格式（缩进、空格、换行）
   - 注释质量
   - 导入组织

2. **设计质量审查**
   - 单一职责原则
   - 开闭原则
   - 依赖倒置原则
   - 接口隔离原则

3. **代码坏味道识别**
   - 重复代码
   - 过长方法/类
   - 过多参数
   - 深层嵌套

4. **安全审查**
   - SQL注入风险
   - XSS漏洞
   - 敏感信息泄露
   - 权限控制缺失

5. **性能问题识别**
   - 数据库N+1查询
   - 不必要的对象创建
   - 低效的算法复杂度
   - 资源泄露

#### 输出
- 代码审查报告
- 问题分级列表（阻塞/警告/建议）
- 改进建议

#### 决策分支
```
审查结果
    │
    ├── 存在阻塞问题 ──► 返回编码Agent修复 ──► 重新审查
    │
    ├── 仅存在警告/建议 ──► 记录问题 ──► 继续下一阶段
    │
    └── 无问题 ──► 继续下一阶段
```

---

### 阶段4：测试生成 (test-generator)

#### 输入
- 审查通过的源代码
- API接口定义

#### 任务
1. **代码分析**
   - 识别公共方法及其职责
   - 分析依赖关系
   - 确定边界条件和异常情况

2. **单元测试生成**
   - 正常场景测试
   - 异常场景测试
   - 边界条件测试

3. **集成测试生成**
   - 数据库交互测试
   - 外部API调用测试
   - 模块间协作测试

4. **端到端测试生成**（关键路径）
   - 主业务流程测试
   - 用户场景测试

5. **测试验证**
   - 运行测试确保通过
   - 检查覆盖率是否达标

#### 输出
- 单元测试代码
- 集成测试代码
- 端到端测试代码
- 测试报告
- 覆盖率统计

#### 覆盖率要求
| 测试类型 | 目标覆盖率 |
|----------|-----------|
| 单元测试 | ≥70% |
| 集成测试 | ≥50% |
| 端到端测试 | 关键路径覆盖 |

## 关键协作点

| 协作点 | 上游Agent | 下游Agent | 传递内容 |
|--------|----------|----------|----------|
| 设计→编码 | architecture-designer | *-coder | 架构文档、接口定义、数据模型 |
| 编码→审查 | *-coder | code-reviewer | 源代码文件、配置文件 |
| 审查→修复 | code-reviewer | *-coder | 问题列表、改进建议 |
| 审查→测试 | code-reviewer | test-generator | 审查通过的代码 |
| 测试→报告 | test-generator | 用户/调用方 | 测试报告、覆盖率统计 |

## 工作流调用示例

### 示例：实现用户管理模块

```yaml
步骤1: 架构设计
  agent: architecture-designer
  input: 
    - 需求：实现用户管理模块（增删改查、权限控制）
    - 技术约束：Java + Spring Boot + MySQL
  output: 架构设计文档

步骤2: 后端编码
  agent: java-coder
  input: 架构设计文档
  output: 后端源代码

步骤3: 前端编码
  agent: frontend-coder
  input: 架构设计文档、后端API定义
  output: 前端源代码

步骤4: 代码审查
  agent: code-reviewer
  input: 前后端源代码
  output: 审查报告
  decision: 如有阻塞问题，返回步骤2/3修复

步骤5: 测试生成
  agent: test-generator
  input: 审查通过的源代码
  output: 测试代码和报告
```

## 异常处理

### 架构设计阶段
- 需求不明确：返回用户澄清需求
- 技术约束冲突：提供替代方案供选择

### 编码阶段
- 架构设计不清晰：与架构设计Agent沟通确认
- 技术实现困难：提出技术方案变更建议

### 审查阶段
- 严重设计问题：返回架构设计阶段重新设计
- 代码质量问题：返回编码阶段修复

### 测试阶段
- 测试无法通过：返回编码阶段修复代码
- 覆盖率不达标：补充测试用例

## 流程中断预防机制

### 预防策略

#### 1. 需求阶段预检
```
用户输入需求
    │
    ▼
┌─────────────────┐
│ 需求完整性检查   │── 缺失必填项？──► 立即提示补充
│ (自动校验)      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 技术可行性预检   │── 技术栈冲突？──► 提前给出替代方案
│ (基于知识库)    │
└────────┬────────┘
         │
         ▼
    进入架构设计
```

#### 2. 编码阶段自检机制
编码Agent提交前自动执行：
- **编译检查**：确保代码可编译
- **单元测试**：基础路径覆盖
- **代码规范**：Linter自动检查
- **安全扫描**：基础规则检查
- **架构对齐**：与设计方案对比

自检不通过 → 自动修复（最多2次）→ 仍失败才提交审查

#### 3. 代码审查分级策略
```
┌─────────────────────────────────────────┐
│           代码审查流水线                 │
├─────────────────────────────────────────┤
│  第一层：自动化检查（秒级）               │
│  - Linter、静态分析、安全扫描            │
│  - 自动修复可修复问题                     │
├─────────────────────────────────────────┤
│  第二层：AI快速审查（分钟级）             │
│  - 仅标记阻塞级问题                       │
│  - 警告级问题记录但不中断                 │
├─────────────────────────────────────────┤
│  第三层：深度审查（需要时触发）            │
│  - 复杂业务场景分析                       │
│  - 性能瓶颈识别                          │
└─────────────────────────────────────────┘
```

### 快速修复机制

| 场景 | 自动修复 | 重试次数 | 仍失败处理 |
|------|---------|---------|-----------|
| 编码自检失败 | 自动修复常见问题 | 2次 | 标记后提交审查 |
| 审查阻塞问题 | 编码Agent按建议修复 | 3次 | 人工介入 |
| 测试失败 | 分析原因并修复 | 2次 | 人工介入 |

### 智能决策减少中断

**可自动决策（不中断流程）**：
- 代码风格不一致 → 自动格式化
- 非核心依赖版本差异 → 使用最新稳定版
- 测试覆盖率略低（核心逻辑已覆盖）→ 记录技术债务
- 补丁级技术选型冲突 → 选择团队最熟悉方案

**必须人工介入**：
- 架构设计重大变更
- 安全漏洞无法自动修复
- 业务流程与需求不符
- 同一问题循环修复超过3次

### 中断率目标
- 简单功能（CRUD）：中断率 < 5%
- 中等复杂度：中断率 < 15%
- 高复杂度：中断率 < 30%

## 流程中断预防机制

### 预防策略

#### 1. 需求阶段强化
- **完整性检查**：自动校验必填项，缺失立即提示
- **技术可行性预检**：基于知识库检查技术栈冲突，提前给出替代方案

#### 2. 架构设计阶段输出标准化
架构设计文档必须包含：
- [ ] 明确的模块边界（减少编码歧义）
- [ ] 详细的接口契约（Request/Response示例）
- [ ] 数据库变更脚本（Flyway格式）
- [ ] 异常处理策略
- [ ] 技术选型理由（避免后续质疑）

#### 3. 编码阶段自检机制
编码Agent提交前自动执行：
- 编译检查
- 单元测试（基础路径）
- 代码规范（Linter）
- 安全扫描（基础规则）
- 与架构设计对比

自检不通过 → 自动修复（最多2次）

#### 4. 代码审查阶段优化
**分级审查策略**：
- **第一层**：自动化检查（Linter、静态分析、安全扫描）
- **第二层**：AI快速审查（仅标记阻塞级问题）
- **第三层**：深度审查（需要时触发）

**阻塞问题预判与规避**：
| 阻塞问题 | 编码阶段规避措施 |
|---------|----------------|
| SQL注入 | 强制使用MyBatis-Plus，禁用字符串拼接SQL |
| NPE风险 | 强制使用Optional，空值检查自动化 |
| 魔法数字 | 常量提取检查，未提取则警告 |
| 长方法 | 方法行数限制（>50行警告，>100行阻塞） |

#### 5. 智能决策减少人工介入
**可自动决策**：
- 技术选型冲突 → 选择团队最熟悉的技术栈
- 代码风格不一致 → 自动格式化
- 测试覆盖率略低（核心逻辑已覆盖）→ 记录技术债务，继续流程
- 非核心依赖版本差异 → 使用最新补丁版本

**必须人工介入**：
- 架构设计重大变更
- 安全漏洞无法自动修复
- 业务流程与需求不符
- 同一问题循环修复超过3次

### 快速修复机制
- 编码Agent自检失败 → 自动修复（2次重试）
- 审查阻塞问题 → 编码Agent自动修复（3次重试）
- 超过重试次数 → 触发人工介入

### 中断率目标
- 简单功能（CRUD）：中断率 < 5%
- 中等复杂度：中断率 < 15%
- 高复杂度：中断率 < 30%

```
         ▼
┌─────────────────┐     ┌─────────────────┐

│ designer        │     │ (API设计)       │
│ (数据库设计)     │     └────────┬────────┘
└────────┬────────┘              │
         │                       ▼
         │              ┌─────────────────┐
         │              │ java-coder      │
         │              │ python-coder    │
         │              │ frontend-coder  │
         │              └────────┬────────┘
         │                       │
         ▼                       ▼
┌─────────────────────────────────────────┐
│         code-reviewer (代码审查)         │
└─────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────┐
│      performance-optimizer (性能优化)    │
│      - 代码性能分析                       │
│      - 数据库查询优化                      │
│      - 缓存策略设计                        │
└─────────────────────────────────────────┘
         │
         ▼

         │
         ▼
      最终交付物
```

### 扩展Agent清单

| Agent | 职责 | 触发条件 |
|-------|------|----------|
| **database-designer** | 数据库表结构设计、索引优化、迁移脚本 | 需要新建/修改数据库时 |
| **api-designer** | RESTful API设计、接口契约定义 | 需要设计新接口时 |
| **performance-optimizer** | 性能分析、优化方案、压测策略 | 性能要求较高或审查发现性能问题时 |
| **devops-engineer** | Docker、CI/CD、K8s、监控配置 | 需要部署配置时 |

### 扩展工作流示例

```yaml
完整电商订单模块开发:
  
  步骤1: architecture-designer
    input: 订单模块需求
    output: 架构设计文档
    
  步骤2: 
    - database-designer
        input: 架构设计文档
        output: 数据库设计、Flyway迁移脚本
    - api-designer
        input: 架构设计文档
        output: API接口定义、OpenAPI文档
        
  步骤3: java-coder
    input: 架构设计、数据库设计、API定义
    output: 后端源代码
    
  步骤4: code-reviewer
    input: 源代码
    output: 审查报告
    
  步骤5: performance-optimizer
    input: 源代码、审查报告
    output: 性能优化建议、优化后代码
    
  步骤6: test-generator
    input: 优化后的源代码
    output: 测试代码
    
  步骤7: devops-engineer
    input: 源代码、测试报告
    output: Dockerfile、CI/CD配置、K8s清单
```

## 扩展工作流（可选阶段）

根据项目需求，可在基础工作流中插入以下可选阶段：

### 阶段A：数据库设计 (database-designer)

**插入位置**：架构设计之后，编码实现之前

**适用场景**：
- 新建业务模块需要数据库表
- 复杂查询需要索引优化
- 需要分库分表设计

**输入**：
- 架构设计文档
- 领域模型定义

**输出**：
- 数据库ER图
- 表结构设计文档
- Flyway迁移脚本
- 索引优化建议

**与编码阶段协作**：
```
architecture-designer
        │
        ▼
database-designer ──► 生成迁移脚本
        │
        ▼
java-coder ──► 使用生成的实体类和Mapper
```

---

### 阶段B：API设计 (api-designer)

**插入位置**：架构设计之后，编码实现之前（可与数据库设计并行）

**适用场景**：
- 前后端分离项目
- 对外提供API接口
- 需要API文档和SDK

**输入**：
- 架构设计文档
- 数据库模型（如有）

**输出**：
- OpenAPI/Swagger文档
- 请求/响应DTO定义
- 接口版本控制策略
- 接口安全设计

**与编码阶段协作**：
```
architecture-designer
        │
        ├──► database-designer
        │
        └──► api-designer ──► 生成DTO和Controller接口
                    │
                    ▼
            java-coder ──► 实现业务逻辑
```

---

### 阶段C：性能优化 (performance-optimizer)

**插入位置**：代码审查之后，测试生成之前

**适用场景**：
- 性能敏感业务（订单、支付）
- 大数据量查询场景
- 高并发接口

**输入**：
- 审查通过的源代码
- 性能指标要求（QPS、RT）

**输出**：
- 性能分析报告
- 优化建议清单
- 优化后的代码
- 压测方案

**决策分支**：
```
性能优化结果
    │
    ├── 性能达标 ──► 继续测试生成
    │
    ├── 轻微问题 ──► 记录技术债务，继续
    │
    └── 严重问题 ──► 返回编码Agent优化 ──► 重新审查
```

---

### 阶段D：DevOps配置 (devops-engineer)

**插入位置**：测试生成之后，最终交付之前

**适用场景**：
- 新服务首次部署
- CI/CD流水线搭建
- 容器化改造

**输入**：
- 源代码和测试代码
- 技术栈信息
- 部署环境要求

**输出**：
- Dockerfile
- CI/CD流水线配置
- Kubernetes部署清单
- 监控告警配置

## 最佳实践

1. **文档驱动**：每个阶段的输出文档是下一阶段的输入
2. **质量门禁**：审查阶段作为质量门禁，阻塞问题必须修复
3. **快速反馈**：发现问题尽快反馈给上游Agent
4. **可追溯性**：保留各阶段的输出，便于问题追溯
5. **并行执行**：前后端编码、数据库和API设计可以并行进行
6. **迭代优化**：复杂功能可以分多轮迭代完成
7. **预防为主**：前置检查减少流程中断
8. **自动修复**：常见问题自动修复，减少人工介入
9. **按需选择**：根据项目复杂度选择必要的Agent，避免过度设计
10. **持续改进**：基于执行反馈优化Agent协作流程
9. **按需扩展**：根据项目需求选择基础或完整工作流
