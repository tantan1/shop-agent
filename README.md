# Shop-Agent 智能客服系统

基于 **RAG（检索增强生成）** 与 **多步骤 Agent 编排** 构建的智能客服平台，面向医院场景提供安全、合规、可追溯的 AI 对话服务。

## 功能概览

### 1. RAG 智能对话

将用户提问向量化后，在 Milvus 知识库中检索相关文档，将检索结果作为上下文注入大模型，生成基于知识库的准确回答，同时返回引用的源文档以保证可追溯性。

### 2. 医院客服多步骤 Agent

针对医疗咨询的高安全要求，设计了四步流水线式 Agent 编排：

- **问题重写** — 将口语化提问改写为标准化医学术语，提升检索精度
- **安全审查** — 检测是否涉及诊断、处方、急危重症等敏感内容，输出风险等级（low / medium / high）
- **知识检索** — 使用嵌入模型 + Milvus 向量检索相关知识文档
- **答案生成** — 基于检索结果与安全审查结论，生成合规、安全的回复

支持普通响应与 **SSE 流式响应** 两种模式，流式接口可实时展示各步骤执行进度。

### 3. 文档知识库管理

支持向 Milvus 向量知识库导入文档，提供单条插入、批量插入和文件上传（`.txt` / `.md` / `.json` / `.csv` 等常见文本格式）三种方式，自动完成文本分块、向量化和入库。

### 4. 企业信息查询

基于 MySQL 提供的企业信息检索服务，支持模糊匹配、精确匹配、地区/行业筛选等多维度查询。

## 关键设计决策

### 双模型策略

项目同时集成了两个大模型，各司其职：

- **通义千问（Qwen）**：通过 OpenAI 兼容模式接入，用于基础 RAG 对话场景，配置简单
- **火山引擎 Doubao**：通过 Ark SDK 直接调用，用于医院 Agent 的多步骤编排，提供低温度（0.3）输出以保证医疗内容的严谨性

### Agent 安全设计

Agent 的安全不是事后可选项，而是内置在流水线中的**第二步**：

- 通过 LLM 识别敏感内容（涉及用药、诊断、处方、儿童、孕妇、急症等），输出结构化风险评估
- LLM 解析失败时，使用**关键词匹配兜底**（保守策略：默认标记为中等风险）
- 高风险问题直接拦截，返回安全警告并引导就医
- 安全审查失败不作为阻塞条件（降级可用），避免审查服务异常导致整体不可用

### Milvus 向量检索策略

- **索引类型**：HNSW（Hierarchical Navigable Small World），适合高召回场景，查询时 `ef` 参数随 `top_k` 动态调整
- **相似度度量**：COSINE，与嵌入模型的归一化输出匹配
- **维度适配**：连接时自动检测集合维度，不匹配则重建

### 单例服务模式

`LLMService`、`EmbeddingService`、`MilvusService` 均采用单例模式，避免重复初始化连接，同时提供懒加载和显式初始化的双路径，兼顾启动速度和运行时健壮性。

### 文本分块策略

使用 LangChain `RecursiveCharacterTextSplitter`，`chunk_size=1000`，`chunk_overlap=200`，以字符级递归分割，确保语义片段完整且相邻块有上下文重叠。

### 统一异常体系

定义了分层异常类：`BusinessException` → `AuthenticationException` / `AuthorizationException` / `NotFoundException` / `ValidationException` / `DatabaseException`，通过 FastAPI 三层异常处理器（业务异常 → HTTP 异常 → 通用异常）统一拦截，前端获得一致的错误响应格式。

### 统一响应格式

```
{ "success": true, "code": 200, "message": "操作成功", "data": {...} }
```

所有接口通过 `success_response()` / `error_response()` 构建标准格式，便于前端统一解析。

### 结构化日志

使用 `structlog` 实现 JSON 格式结构化日志，关键设计：

- 请求级中间件自动记录方法、路径、耗时、状态码
- 业务事件日志记录操作名称、成功状态、相关参数
- **日志脱敏**：API Key 仅记录前 8 位，用户消息仅记录前 20 字符，避免敏感信息泄露

### 异步数据库访问

采用 SQLAlchemy 2.0 异步引擎 + `aiomysql` 驱动，连接池预检查（`pool_pre_ping=True`）和定期回收（`pool_recycle=3600`），通过 FastAPI `Depends` 依赖注入管理会话生命周期。


### 流式响应

医院客服 Agent 流式接口采用 **SSE（Server-Sent Events）** 协议，按步骤推送事件（`step_start` → `step_complete` → `content` → `done`），让前端实时感知 Agent 内部进展。

### 基础设施容器化

通过 `docker-compose.yml` 一键编排 Milvus 全套依赖（etcd + MinIO + Milvus Standalone）以及 Redis 缓存服务，降低本地开发环境搭建成本。
